Case study 01

Enterprise Data Discovery Platform

General Motors IT · Information Architecture & UX Lead
The problem

No single view of enterprise data assets

ETL, storage, governance, and BI teams each maintained separate catalogs with no cross-team visibility. Finding a dataset meant knowing the right person to ask.

Finding data took days or weeks

Users relied on word-of-mouth. When people left, the knowledge went with them.

Access control complicated discovery

Search results had to respect data security tiers — Sarbanes-Oxley, PII, HR, R&D — without exposing the existence of assets a user wasn't permitted to see.

Role & outcomes
  1. Led IA and UX for the full platform — taxonomy design, metadata standards, search architecture, and access-aware UI
  2. Reduced data discovery from days or weeks to minutes for thousands of global users
  3. Coordinated metadata collection across ETL, governance, BI, and cloud storage teams
  4. Implemented security-aware search: results filtered by user permissions without revealing restricted assets exist
Domain model — core object relationships
Key IA challenge

Search results had to be access-aware — returning only what a user was permitted to see, without revealing that restricted assets existed at all.

Case study 02

DevSecOps Compliance Platform

General Motors IT · UX & Information Architecture
The problem

Fragmented tooling

Testing, deployment status, and milestone tracking lived in separate systems. No single view of where a deployment stood in the pipeline.

Review board prep was manual and slow

PMs assembled compliance packages by hand before each review meeting. Backout plans, contact info, and sign-offs existed in isolation from the deployments they governed.

Compliance docs were orphaned from deployments

Each gate required specific documentation, but there was no structural link between the docs and the gate — or the deployment.

Role & outcomes
  1. Designed the unified platform IA — staged pipeline model, gate structure, and compliance doc linkage
  2. Eliminated manual review board prep: PMs no longer assembled compliance packages by hand
  3. Gave compliance reviewers a single view of each deployment's status, linked docs, and QA results
  4. Platform became the operational standard for DevSecOps workflow tracking across GM IT
Information architecture — staged deployment pipeline
Key design problem

Each gate's required docs lived in separate systems with no structural link to the deployment or gate they governed. The platform made those relationships explicit and navigable.

Case study 03

Product Resource Forecasting

General Motors IT · UX Strategy & IA
The problem

Planning ran on static spreadsheets

Headcount and resource needs were projected manually, with no way to model multiple scenarios or compare assumptions.

Decision-makers had no shared view

Finance, HR, and product leaders each worked from different versions of headcount data with no reconciliation point.

Scenarios couldn't be compared

Leadership needed to evaluate multiple forecast models side-by-side, but the platform had to surface the right scenario at the right decision point.

Role & outcomes
  1. Designed UX for scenario input, filtering, and comparison views used by finance and product leadership
  2. Defined the IA connecting prediction model outputs to filterable, role-appropriate scenario lists
  3. Replaced manual spreadsheet planning with a shared, real-time view reconciling Finance, HR, and product inputs
  4. Reduced scenario review prep time; enabled leadership to evaluate forecasts side-by-side in a single session
User decision flow — submit · filter · select scenario
Key IA decision

Scenarios needed to be filterable objects, not report outputs — which required defining the data model before any UI work could begin.

Approach

Diagnose before designing

Every project starts with surfacing misaligned assumptions. "Shared vocabulary doesn't mean shared meaning" — the reference interview trained me to find the gap between what people say they need and what the problem actually is.

Define the objects before the interface

Domain modeling, taxonomy design, and metadata standards before wireframes. The structure is the decision.

Cross-functional by default

Embedded with dev, business, and governance stakeholders — not handed off to them. Influence through clarity of craft, not organizational hierarchy.

Documentation that survives turnover

Systems designed so the logic is clear enough that people use the docs to train the next person.
